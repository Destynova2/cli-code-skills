# Watermark Lifecycle — Preservation, CI Integration, Tamper-Proofing

> **When to read:** After Phase 3 (proof artifacts generated). This file covers how to make watermarks survive normal development, detect accidental degradation, and resist deliberate removal.

---

## The problem: your own devs will kill your watermarks

The biggest threat to your watermarks is not an adversary — it's your own team:
- A dev replaces 6735 with 8192 "because power of two is cleaner"
- A refactor removes the canary guard clause "because it's redundant"
- A clean-code pass normalizes JSON field ordering to alphabetical
- A dep update changes a Fermat-prime constant in a formula

**Rule:** watermarks that aren't monitored are watermarks that decay.

---

## Layer 1: Pre-commit hook — fragile watermark sentinel

A hidden pre-commit hook that verifies critical watermark values haven't been accidentally changed.

### Implementation

Generate `{project}/.watermark/hooks/check-watermarks.sh`:

```bash
#!/usr/bin/env bash
# Watermark integrity check — runs as pre-commit hook
# Generated by cli-watermark. DO NOT remove.
set -euo pipefail

WATERMARK_DIR="$(git rev-parse --show-toplevel)/.watermark"
CODEBOOK="${WATERMARK_DIR}/codebook.json"

[ -f "$CODEBOOK" ] || exit 0  # No codebook = no watermarks to check

FAILURES=0

# Check L1 constants are still present with expected values
check_constant() {
  local file="$1" pattern="$2" name="$3"
  if ! grep -q "$pattern" "$file" 2>/dev/null; then
    echo "WARNING: watermark constant $name may have been modified in $file"
    FAILURES=$((FAILURES + 1))
  fi
}

# Check L3 canaries are still present
check_canary() {
  local file="$1" pattern="$2" name="$3"
  if ! grep -q "$pattern" "$file" 2>/dev/null; then
    echo "WARNING: watermark canary $name may have been removed from $file"
    FAILURES=$((FAILURES + 1))
  fi
}

# --- Auto-generated checks from codebook.json ---
# (cli-watermark Phase 3 generates these lines)
{GENERATED_CHECKS}

if [ "$FAILURES" -gt 0 ]; then
  echo ""
  echo "=== WATERMARK INTEGRITY: $FAILURES issue(s) detected ==="
  echo "If you intentionally changed these values, run:"
  echo "  cli-watermark --refresh"
  echo "to update the codebook and re-derive the watermarks."
  # WARNING only, don't block the commit — dev velocity matters
  # Change to 'exit 1' for strict mode
  exit 0
fi
```

### Installation

```bash
# Add to .git/hooks/pre-commit (or integrate with husky/pre-commit framework)
cat >> .git/hooks/pre-commit << 'HOOK'
# Watermark integrity check
bash .watermark/hooks/check-watermarks.sh
HOOK
chmod +x .git/hooks/pre-commit
```

For projects using the `pre-commit` framework:

```yaml
# .pre-commit-config.yaml
repos:
  - repo: local
    hooks:
      - id: watermark-check
        name: Watermark integrity
        entry: bash .watermark/hooks/check-watermarks.sh
        language: system
        always_run: true
        pass_filenames: false
```

### Why WARNING, not BLOCK

Blocking commits for watermark issues kills dev velocity and creates resentment. The hook:
- WARNS in the terminal so the dev notices
- Does NOT block the commit
- Logs to `.watermark/integrity.log` for audit
- The CI pipeline (below) is the actual gate

## Layer 2: Hidden CI job — the silent guardian

A CI job that runs on every push but is **not visible** in the main pipeline status. It writes to a private artifact, not to the PR checks.

### GitHub Actions (hidden job)

Generate `{project}/.github/workflows/watermark-verify.yml`:

```yaml
name: integrity
on:
  push:
    branches: [main, develop]
  pull_request:
    branches: [main]

# This workflow does NOT block PRs — it's a silent observer
jobs:
  verify:
    runs-on: ubuntu-latest
    # Only runs if the watermark codebook exists
    if: hashFiles('.watermark/codebook.json') != ''
    steps:
      - uses: actions/checkout@v4

      - name: Verify watermark constants
        run: bash .watermark/hooks/check-watermarks.sh 2>&1 | tee /tmp/wm-report.txt

      - name: Verify test vectors
        run: |
          if [ -f .watermark/vectors.json ]; then
            # Run the project's test suite filtered to watermark vectors
            # (project-specific — adapt the command)
            echo "Test vector verification: TODO — adapt to project test runner"
          fi

      - name: Generate integrity report
        run: |
          echo "## Watermark Integrity — $(date -Is)" > /tmp/wm-summary.md
          echo "" >> /tmp/wm-summary.md
          if grep -q "WARNING" /tmp/wm-report.txt; then
            echo "**STATUS: DEGRADED** — some watermark signals have changed" >> /tmp/wm-summary.md
            echo "" >> /tmp/wm-summary.md
            grep "WARNING" /tmp/wm-report.txt >> /tmp/wm-summary.md
          else
            echo "**STATUS: INTACT** — all watermark signals verified" >> /tmp/wm-summary.md
          fi

      - name: Upload report (private artifact, expires in 7 days)
        uses: actions/upload-artifact@v4
        with:
          name: watermark-integrity
          path: /tmp/wm-summary.md
          retention-days: 7
```

### Why silent, not blocking

1. **OpSec**: a visible "watermark check" job tells the adversary watermarks exist
2. **Dev velocity**: devs should never be blocked by watermark CI
3. **Audit trail**: the artifact is there for forensics, invisible to PRs

### GitLab CI (hidden job)

```yaml
watermark-verify:
  stage: .post          # Runs after all other stages
  rules:
    - if: $CI_COMMIT_BRANCH == "main"
    - if: $CI_PIPELINE_SOURCE == "merge_request_event"
  allow_failure: true   # Never blocks the pipeline
  script:
    - bash .watermark/hooks/check-watermarks.sh 2>&1 | tee wm-report.txt
  artifacts:
    paths: [wm-report.txt]
    expire_in: 7 days
    when: always
```

## Layer 3: Entangled constants (Collberg/Thomborson technique)

Make watermark values **load-bearing** in the code, so removing them breaks something.

### Technique: derived test expectations

Write tests whose expected values are derived from watermark constants:

```rust
#[test]
fn test_cache_behavior() {
    let cache = RouteCache::new();  // uses ROUTE_CACHE_CAPACITY = 6735

    // Fill to 90% capacity
    for i in 0..(6735 * 9 / 10) {
        cache.insert(format!("key_{i}"), Route::default());
    }

    // This assertion depends on the exact capacity value
    assert_eq!(cache.len(), 6061);  // 6735 * 0.9 = 6061.5, truncated
    assert!(!cache.is_full());

    // One more push to trigger eviction at exactly this capacity
    for i in 0..675 {
        cache.insert(format!("extra_{i}"), Route::default());
    }
    assert_eq!(cache.eviction_count(), 1);  // eviction at 6735, not at 6736
}
```

If someone changes `ROUTE_CACHE_CAPACITY` from 6735 to 8192, this test fails. The test looks like a normal capacity test — nothing suspicious.

### Technique: cross-referencing constants

Make two watermark constants depend on each other:

```rust
const ROUTE_CACHE_CAPACITY: usize = 6735;
const AUDIT_RING_SIZE: usize = 11134;

// Somewhere in the code, an assertion that looks like a sanity check:
debug_assert!(
    AUDIT_RING_SIZE > ROUTE_CACHE_CAPACITY,
    "audit ring must be larger than route cache"
);

// A test that checks the relationship:
#[test]
fn test_buffer_sizing_invariant() {
    // Ring must hold at least 1.5x the cache to avoid backpressure
    assert!(AUDIT_RING_SIZE as f64 / ROUTE_CACHE_CAPACITY as f64 > 1.5);
    assert!(AUDIT_RING_SIZE as f64 / ROUTE_CACHE_CAPACITY as f64 < 2.0);
}
```

Changing either constant independently breaks the invariant test.

### Technique: behavioral test lock

For L5 behavioral markers, write integration tests that assert the exact behavior:

```rust
#[test]
fn test_response_field_ordering() {
    let resp = route_request(test_input());
    let json = serde_json::to_string(&resp).unwrap();

    // Assert field ordering (model before id before ts)
    let model_pos = json.find("\"model\"").unwrap();
    let id_pos = json.find("\"id\"").unwrap();
    let ts_pos = json.find("\"ts\"").unwrap();
    assert!(model_pos < id_pos, "model must come before id");
    assert!(id_pos < ts_pos, "id must come before ts");
}

#[test]
fn test_score_precision() {
    let resp = route_request(canonical_vector_42());
    // Exact precision is part of the watermark
    assert_eq!(format!("{:.5}", resp.score), "0.73418");
}
```

These tests look like normal API contract tests. But they lock the L5 watermark in place.

## Layer 4: Release-time audit

At each release, run a full watermark audit. This is the `/cli-watermark --audit` mode.

### Audit checklist

```
=== WATERMARK AUDIT — Release v{VERSION} ===

[ ] L1 Constants — all HMAC-derived values present at expected locations
    Method: grep codebook constants in source, compare with derivation
[ ] L3 Canaries — all paper-town patterns present
    Method: grep canary signatures in source
[ ] L4 Algorithm — fingerprint constants present in formulas
    Method: grep rare primes / specific patterns
[ ] L5 Behavior — test vectors still produce expected output
    Method: run canonical test vectors, compare 5-decimal precision
[ ] L6 CFG — control flow topology matches documented structure
    Method: compare branch ordering against codebook
[ ] L7 Vectors — published vectors match current implementation
    Method: run all published vectors
[ ] L8 Commitment — generate new commitment for this release
    Method: derive tree hash, commit, OTS stamp

Results:
  Intact:    N/8 layers
  Degraded:  N/8 layers (list which + what changed)
  Missing:   N/8 layers (removed or never applied)

Action required:
  - Re-derive degraded layers
  - Update codebook.json
  - New L8 commitment for this release
```

### Integration with cli-cycle

When `/cli-cycle` runs, if `.watermark/codebook.json` exists, it should automatically include watermark integrity in the scorecard:

```
| Watermark integrity | 7/8 layers intact | L1 degraded | cli-watermark |
```

## Layer 5: Guard network (inspired by Collberg sentinel network)

Distribute N small verification checks across the codebase. Each guard:
- Looks like a normal `debug_assert!` or test
- Verifies one watermark signal
- Is not co-located with the watermark it guards

```rust
// In src/router/mod.rs — guards L1 constant in src/cache/config.rs
debug_assert!(
    crate::cache::ROUTE_CACHE_CAPACITY < 10000,
    "cache capacity out of expected range"
);

// In src/audit/ring.rs — guards L4 constant in src/router/scoring.rs
#[cfg(test)]
mod guards {
    #[test]
    fn scoring_uses_expected_normalization() {
        // This test "documents" the scoring formula but actually guards L4
        let score = crate::router::score_route(1000.0, 0.5, 1.0, 0.5);
        // The expected value depends on the Fermat prime F4 = 65537
        assert!((score - 0.01525).abs() < 0.0001);  // 1000/65537 * 1.0
    }
}
```

**Distribution rules:**
- 1 guard per watermark layer minimum
- Guards in different files from what they protect
- Guards look like normal invariant tests or debug assertions
- No file named `watermark_guard` or `integrity_check` — they are embedded

## Summary: the 5-layer defense

```
┌─────────────────────────────────────────────────┐
│  Layer 5: Guard network (in the code)            │
│  Distributed debug_assert + tests that verify    │
│  watermark values indirectly. Removal = test fail│
├─────────────────────────────────────────────────┤
│  Layer 4: Release audit (/cli-watermark --audit) │
│  Full check of all 8 watermark layers.           │
│  Run before every tag/release.                   │
├─────────────────────────────────────────────────┤
│  Layer 3: Entangled constants (tests)            │
│  Tests whose expected values derive from the     │
│  watermark. Change the constant = test fails.    │
├─────────────────────────────────────────────────┤
│  Layer 2: Hidden CI job (silent)                 │
│  Runs on every push. Never blocks. Writes to     │
│  private artifact for forensics.                 │
├─────────────────────────────────────────────────┤
│  Layer 1: Pre-commit hook (warning)              │
│  Developer sees a warning if they changed a      │
│  watermark value. Non-blocking.                  │
└─────────────────────────────────────────────────┘
```

**The hierarchy is intentional:**
- Pre-commit = fast feedback for the dev (seconds)
- CI = async verification (minutes)
- Entangled tests = break on change (test suite)
- Guards = distributed, hard to find and remove all
- Release audit = comprehensive, manual sign-off

## OpSec considerations

1. **The pre-commit hook is the only visible element.** Name it generically: "integrity check" not "watermark check"
2. **The CI job should not appear in PR checks.** Use `allow_failure: true` (GitLab) or don't add it to required checks (GitHub)
3. **Guard tests should not be in a `watermark/` directory.** Scatter them across the test suite
4. **The `.watermark/` directory should be gitignored** except for `PROOF.md` and `vectors.json` (if you choose to publish them)
5. **Never mention watermarks in commit messages.** Use neutral language: "adjust buffer sizing", "update scoring constants"
